# Requirements: JavaScript Notebook — LLM Integration

## 1. Обзор проекта

**JavaScript Notebook** — это веб-приложение в стиле Jupyter Notebook для написания и выполнения JavaScript-кода прямо в браузере, с поддержкой текстовых блоков, синхронизацией через бэкенд и генерацией кода с помощью LLM.

### 2.2 Хранение данных

| Слой        | Технология       | Назначение                                  |
|-------------|------------------|---------------------------------------------|
| Frontend    | IndexedDB        | Офлайн-хранение всех ноутбуков              |
| Backend     | PostgreSQL/SQLite | Синхронизированные копии ноутбуков          |
| Формат      | JSON             | Структура ноутбука (см. раздел 4)           |

### 2.3 Где живёт LLM?

**LLM-запросы проксируются через бэкенд.**

Причины:
- API-ключ не должен быть виден на клиенте
- Возможность логировать и ограничивать запросы (rate limiting)
- Централизованная замена модели без деплоя фронтенда

Схема вызова:
```
Frontend → POST /api/llm/generate → Backend → Anthropic/OpenAI API → Backend → Frontend
```

## 3. Требования к LLM-интеграции

### 3.1 Функциональные требования

| ID     | Требование                                                                                       |
|--------|--------------------------------------------------------------------------------------------------|
| LLM-01 | Пользователь создаёт текстовый блок с описанием задачи и нажимает кнопку **Generate Code**       |
| LLM-02 | Система отправляет содержимое текстового блока на бэкенд                                         |
| LLM-03 | Бэкенд формирует промпт и вызывает LLM API                                                       |
| LLM-04 | LLM возвращает код, который вставляется в новый или существующий code-блок под текстовым         |
| LLM-05 | Пользователь может отредактировать сгенерированный код перед выполнением                         |
| LLM-06 | Генерация запускается только явно (кнопкой), не автоматически                                    |
| LLM-07 | Поддерживается контекст: в промпт можно опционально включить соседние блоки ноутбука             |

### 3.2 Нефункциональные требования

| ID     | Требование                                                                                          |
|--------|-----------------------------------------------------------------------------------------------------|
| LLM-NF-01 | Время ответа LLM не должно превышать 30 секунд; при превышении — таймаут с сообщением об ошибке |
| LLM-NF-02 | Стриминг ответа (SSE/WebSocket) для отображения кода по мере генерации                          |
| LLM-NF-03 | Rate limiting: не более 20 запросов к LLM в минуту на пользователя                              |
| LLM-NF-04 | Логирование всех LLM-запросов (prompt, model, tokens, latency) на бэкенде                       |
| LLM-NF-05 | API-ключ хранится только на сервере (env variable), никогда не передаётся клиенту               |

### 3.3 Формат промпта

```
System:
  Ты — помощник, который пишет чистый JavaScript-код.
  Возвращай ТОЛЬКО код без объяснений и markdown-блоков.
  Код должен работать в браузерной sandbox-среде без Python API.

User:
  Контекст ноутбука (опционально):
  [предыдущие блоки]

  Задача:
  [содержимое текстового блока пользователя]
```

```json
      "generatedByLLM": true
    }
  ]

```

## 5. Тесты

### 5.1 Unit-тесты (Frontend)

| ID     | Компонент      | Сценарий                                                       | Ожидаемый результат                          |
|--------|----------------|----------------------------------------------------------------|----------------------------------------------|
| UT-F-01 | Cell Store    | Добавление code-блока                                          | Блок появляется в списке cells               |
| UT-F-02 | Cell Store    | Удаление блока                                                 | Блок исчезает, порядок сохраняется           |
| UT-F-03 | Cell Store    | Перемещение блока (drag & drop)                               | Порядок cells обновляется корректно          |
| UT-F-04 | JS Runtime    | Выполнение `2 + 2`                                             | Output: `4`                                  |
| UT-F-05 | JS Runtime    | Выполнение кода с синтаксической ошибкой                      | Output: сообщение об ошибке, без краша UI    |
| UT-F-06 | JS Runtime    | Выполнение `console.log('test')`                               | Output: `test`                               |
| UT-F-07 | LLM Client    | Успешный запрос — ответ с кодом                               | Код вставляется в новый code-блок            |
| UT-F-08 | LLM Client    | Таймаут запроса (> 30s)                                        | Показывается ошибка, блок не создаётся       |
| UT-F-09 | Serializer    | Сериализация ноутбука в JSON                                   | JSON соответствует схеме (раздел 4)          |
| UT-F-10 | Serializer    | Десериализация корректного JSON                                | Все блоки восстановлены                      |
| UT-F-11 | IndexedDB     | Сохранение и чтение ноутбука офлайн                            | Данные идентичны до и после                 |

### 5.2 Unit-тесты (Backend)

| ID     | Сервис          | Сценарий                                                      | Ожидаемый результат                          |
|--------|-----------------|---------------------------------------------------------------|----------------------------------------------|
| UT-B-01 | Auth           | Регистрация с валидными данными                               | Создан пользователь, возвращён JWT           |
| UT-B-02 | Auth           | Вход с неверным паролем                                       | 401 Unauthorized                             |
| UT-B-03 | Auth           | Запрос с истёкшим JWT                                         | 401, токен отклонён                          |
| UT-B-04 | Sync           | Сохранение ноутбука (POST /api/notebooks)                     | 201, ноутбук в БД                            |
| UT-B-05 | Sync           | Получение списка ноутбуков пользователя                       | Только ноутбуки данного пользователя         |
| UT-B-06 | Sync           | Обновление ноутбука с устаревшим `updatedAt`                  | 409 Conflict                                 |
| UT-B-07 | LLM Proxy      | Успешный проброс запроса к LLM API                            | Возвращён сгенерированный код                |
| UT-B-08 | LLM Proxy      | LLM API недоступен                                            | 502 Bad Gateway, сообщение об ошибке         |
| UT-B-09 | LLM Proxy      | Rate limit превышен (> 20 req/min)                            | 429 Too Many Requests                        |
| UT-B-10 | LLM Proxy      | API-ключ отсутствует в env                                    | 500, сервер не запускается / логирует ошибку |

### 5.3 Интеграционные тесты

| ID     | Сценарий                                                                                      | Ожидаемый результат                                             |
|--------|-----------------------------------------------------------------------------------------------|------------------------------------------------------------------|
| IT-01  | Пользователь регистрируется → логинится → создаёт ноутбук → синхронизирует                   | Ноутбук сохранён в БД и в IndexedDB                            |
| IT-02  | Пользователь создаёт text-блок с описанием → нажимает Generate → получает code-блок         | Code-блок с кодом появляется под text-блоком                   |

| IT-04  | Два пользователя синхронизируют разные ноутбуки одновременно                                 | Нет конфликтов между ноутбуками разных пользователей           |
| IT-05  | Выполнение кода с бесконечным циклом                                                          | Runtime прерывается по таймауту (5s), UI не зависает           |

## 6. Дополнительные фичи (опциональные)

- **TypeScript-поддержка** — игнорирование типов (как в Bun), выполнение TS как JS
- **Экспорт ноутбука** — скачать как `.json` или `.js` файл
- **Shared notebooks** — публичные ссылки на ноутбук (read-only)
- **Тёмная/светлая тема** — переключатель
- **Markdown-рендеринг** — text-блоки рендерятся как Markdown
- **Горячие клавиши** — `Ctrl+Enter` для выполнения, `Shift+Enter` для создания блока
